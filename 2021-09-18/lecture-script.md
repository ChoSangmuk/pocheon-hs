## 11장. SVN vs Git
```
눈치가 빠른 분이라면 아셨겠지만 Git이 등장하기 이전에도 소스 코드 관리 시스템,
편의상 SCM이라고 하겠습니다. SCM은 있었습니다.
다만 이 시스템들은 치명적인 단점을 가지고 있었습니다.
 
바로 중앙 집중의 소스 관리 시스템이라는 것 입니다.
Git 이전 버젼의 SCM 대표적으로 SVN은 모든 소스는 중앙에 저장하였고, 
서버와 클라이언트와 유사한 개념으로 동작하였기 때문에 
제한적으로 개인의 개발 이력을 관리해야 했으며,
충돌이 일어날 확률이 매우 높았습니다.

만약 SVN 서버가 펑하고 터지면 SVN을 다시 설치해야하고 이력도 복원해야합니다.
하지만 git은 똑같은 내용의 저장소를 여러 곳에 저장할 수 있으며,
개발자는 브랜치라는 개념을 통해 각자의 버젼을 관리함으로 
서버 폭파와 충돌 면에서 많은 유리함이 있습니다.

그림 상에는 SVN 과 다를것 없어 보이지만 위의 git 저장소는 
개인이 로컬에 보유한 저장소와 동등한 위치로 보면됩니다.
다시 말하면 svn은 명확하게 서버, 클라이언트 구조이지만 
git의 모든 저장소는 동등한 지위를 가지며 
사람에 의해 그 의미가 나눠지는 것 뿐이라고 생각하면 됩니다.

어찌되었건 여러 버전을 관리해야하고
기존의 svn과 같은 구형 SCM에 불편함이 있다면
SCM으로 Git을 써야하지 않을까요?
지금 당장은 아니더라도 사용법 정도는 알고 있어야하지 않을까요??
```
